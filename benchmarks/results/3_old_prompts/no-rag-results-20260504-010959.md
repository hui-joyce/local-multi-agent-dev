### Assumptions
1.  **Concurrency**: The implementation assumes a multi-threaded environment. Therefore, the `TokenBucket` class uses a `threading.Lock` to ensure thread safety when acquiring tokens.
2.  **Time Source**: The implementation uses `time.monotonic()` for timing. This is preferred over `time.time()` for rate limiting as it is not affected by system clock adjustments (e.g., NTP synchronization), preventing race conditions in token refill calculations.
3.  **Token Precision**: Tokens are stored as floats to allow for fractional accumulation (e.g., 0.5 tokens), which provides smoother rate limiting than integer-only buckets.
4.  **Usage Pattern**: The limiter is designed to be instantiated once per resource (e.g., one limiter for API calls, another for database writes) and reused across threads.

### Implementation

```python
import threading
import time
from typing import Optional, Tuple

class TokenBucket:
    """
    A thread-safe Token Bucket rate limiter.
    
    Logic:
    - The bucket has a maximum capacity (max_tokens).
    - Tokens are added at a constant rate (refill_rate) per second.
    - Requests consume tokens. If tokens are available, the request is allowed.
    - If tokens are exhausted, the request must wait until enough tokens are refilled.
    
    Thread Safety:
    - Uses a threading.Lock to protect the internal state during acquire operations.
    """

    def __init__(self, max_tokens: float, refill_rate: float):
        """
        Initialize the token bucket.
        
        Args:
            max_tokens (float): The maximum number of tokens the bucket can hold.
            refill_rate (float): The number of tokens added per second.
        """
        self.max_tokens = max_tokens
        self.refill_rate = refill_rate
        self.tokens = max_tokens  # Start with a full bucket for burst capability
        self.last_refill_time = time.monotonic()
        self._lock = threading.Lock()

    def _refill(self) -> None:
        """
        Internal method to refill tokens based on elapsed time.
        Must be called while holding the lock.
        """
        now = time.monotonic()
        elapsed = now - self.last_refill_time
        tokens_to_add = elapsed * self.refill_rate
        
        self.tokens = min(self.max_tokens, self.tokens + tokens_to_add)
        self.last_refill_time = now

    def acquire(self, tokens: float = 1.0) -> Tuple[bool, float]:
        """
        Attempt to acquire the specified number of tokens.
        
        Args:
            tokens (float): Number of tokens to acquire. Defaults to 1.0.
            
        Returns:
            Tuple[bool, float]: 
                - True if tokens were acquired.
                - False if not enough tokens are available.
                - The second value is the estimated wait time in seconds if denied.
        """
        with self._lock:
            # Refill tokens based on time passed since last check
            self._refill()

            if self.tokens >= tokens:
                self.tokens -= tokens
                return True, 0.0
            else:
                # Calculate how long we need to wait to get the required tokens
                tokens_needed = tokens - self.tokens
                wait_time = tokens_needed / self.refill_rate
                return False, wait_time

    def wait(self, tokens: float = 1.0) -> None:
        """
        Block the current thread until the specified number of tokens are available.
        This is a blocking version of acquire.
        
        Args:
            tokens (float): Number of tokens to acquire. Defaults to 1.0.
        """
        while True:
            acquired, wait_time = self.acquire(tokens)
            if acquired:
                return
            # Sleep for a small fraction of the wait time to avoid busy-waiting
            # Using a small sleep prevents high CPU usage while waiting
            sleep_time = min(wait_time, 0.01)
            time.sleep(sleep_time)

# Example Usage
if __name__ == "__main__":
    # Allow 10 requests per second, with a burst capacity of 20
    limiter = TokenBucket(max_tokens=20, refill_rate=10)
    
    # Simulate a burst of 25 requests
    for i in range(25):
        allowed, wait = limiter.acquire()
        if allowed:
            print(f"Request {i+1}: Allowed")
        else:
            print(f"Request {i+1}: Denied (Wait {wait:.2f}s)")
```

### Key Design Decisions Explained
*   **Float vs Int**: Using floats allows for sub-second precision in rate limiting (e.g., 1.5 requests per second), which is often required for API throttling.
*   **`time.monotonic()`**: This ensures that if the system clock is adjusted (e.g., via NTP or manual change), the rate limiter's internal clock does not jump, preventing the bucket from suddenly gaining or losing tokens unexpectedly.
*   **Locking Strategy**: A single `threading.Lock` protects the entire `acquire` logic. While this introduces slight contention, it is the simplest and most robust way to prevent race conditions in Python's GIL-less context for float arithmetic.
*   **Non-blocking `acquire`**: The `acquire` method returns immediately with a boolean and a wait time estimate. This allows the caller to make a decision (e.g., queue the request or reject it) without blocking the thread, which is crucial for high-concurrency servers.
*   **Blocking `wait`**: The `wait` method is provided for scenarios where the caller can afford to block (e.g., a background worker thread) and wants to proceed as soon as a token is available.

```text
### Reverse Engineering Analysis

#### 1. High-Level Intent
The function implements a **state-machine based protocol parser** combined with a **custom checksum verification**. Its primary goal is to validate that an input stream matches a specific binary structure (delimited by specific byte values) and possesses a cryptographic or integrity signature (the checksum). If both conditions are met, it triggers a `privileged_operation`.

**Likely Scenario:** This code protects a specific command or data payload from being executed. The input must be a valid packet containing a Header, Body, and End marker, and the calculated checksum must match a hardcoded constant (`0xBEEF`).

#### 2. State Machine Logic Flow
The parser operates as a finite state machine (FSM) with four states: `INIT`, `HEADER`, `BODY`, and `DONE`.

*   **State Transitions:**
    *   `INIT` $\to$ `HEADER`: Triggered when the first byte (`b`) is `0x7B` (ASCII `{`).
    *   `HEADER` $\to$ `BODY`: Triggered when the current byte is `0x3A` (ASCII `:`).
    *   `BODY` $\to$ `DONE`: Triggered when the current byte is `0x7D` (ASCII `}`).
    *   *Note:* The transitions are strictly sequential. If a byte appears out of order (e.g., `}` before `:`), the state remains unchanged, potentially causing the parser to ignore that byte or fail later.

*   **Termination Condition:**
    *   The loop continues until `idx` reaches the end of the input.
    *   The function only returns `OK` if the final state is `DONE`. If the input ends while in `INIT`, `HEADER`, or `BODY`, it returns `ERR_FORMAT`.

#### 3. Checksum Calculation & Hidden Assumptions
The checksum logic is the core security mechanism, but it contains several critical hidden assumptions and potential flaws:

*   **The Formula:**
    `checksum = (checksum + ((b XOR key[idx % len(key)]) * 17)) & 0xFFFF`
    *   **Byte-wise Processing:** The checksum is updated for every single byte in the input, not just the payload.
    *   **Key Rotation:** It uses a repeating key (`key`) to XOR with the input bytes. The index into the key is determined by the global loop index (`idx`), not the position within the payload.
    *   **Multiplier:** Every byte is multiplied by `17` before addition.
    *   **Overflow Handling:** The result is masked with `0xFFFF` (16-bit unsigned integer) at every step. This prevents carry propagation beyond 16 bits, effectively making the checksum calculation **non-associative** across the whole stream. The final result depends heavily on the specific byte order and the specific values of the key.

*   **Hidden Assumptions:**
    1.  **Key Length:** The code assumes `len(key)` is non-zero. If `key` is empty, `idx % 0` will crash.
    2.  **Input Length:** The code assumes `len(input)` is non-zero. If input is empty, the loop doesn't run, `state` stays `INIT`, and it returns `ERR_FORMAT`.
    3.  **Byte Range:** The logic assumes `input` contains bytes (0-255). If `input` contains larger integers (e.g., 32-bit words), the `b` variable might need masking (`& 0xFF`), though the pseudo-code implies `b` is already a byte.
    4.  **State Independence:** The checksum calculation happens **regardless of the current state**. This is a significant vulnerability. Malicious actors can inject bytes in the `INIT` or `HEADER` states that contribute to the final checksum, potentially allowing them to "poison" the checksum calculation to match `0xBEEF`.

#### 4. Abuse Cases & Vulnerabilities

*   **Truncation Attack (Format Error):**
    An attacker can send a packet with a valid Header (`{`) and Body content, but omit the closing delimiter (`}`). The state will remain `BODY`, and the function returns `ERR_FORMAT`. This is a standard defense against incomplete packets.

*   **Checksum Forgery (The Critical Flaw):**
    Because the checksum is calculated incrementally and masked (`& 0xFFFF`), an attacker can manipulate the input to force the final checksum to be `0xBEEF`.
    *   *Scenario:* The attacker knows the `key` and the `input` structure. They can calculate the "expected" checksum for the first $N$ bytes. If the calculated checksum is not `0xBEEF`, they can append a specific number of padding bytes (or modify existing bytes if the state machine allows it) to shift the checksum value into the target range.
    *   *Exploiting the Mask:* Since the operation is `& 0xFFFF`, the checksum wraps around. An attacker can add a specific byte value $X$ such that `(current_checksum + (X * 17)) % 65536 == 0xBEEF`. Since there are 256 possible byte values, and the multiplier is 17, the attacker can likely find a byte that shifts the checksum into the desired range, provided the input length allows for enough adjustment.

*   **State Injection:**
    The state machine is rigid. If an attacker sends `0x7D` (the `}` character) as the very first byte, the state remains `INIT` (because the condition `state == INIT and b == 0x7B` fails). The parser will then ignore this byte and move to the next one. This could be used to hide malicious data at the start of the stream, though the final `DONE` check usually prevents this unless the rest of the stream is perfectly crafted.

*   **Key Dependency:**
    The security relies entirely on the secrecy of `key`. If `key` is static and known (e.g., hardcoded in the client), the checksum is trivially breakable. The attacker can pre-calculate the required input bytes to generate the `0xBEEF` signature.

#### 5. Recommendations for Hardening

1.  **Fix the Checksum Calculation:**
    *   Do not calculate the checksum over the delimiters (`{`, `:`, `}`). Calculate it only over the `BODY` section.
    *   Remove the per-step masking (`& 0xFFFF`) if the checksum algorithm requires carry propagation (e.g., CRC32). If a simple additive checksum is intended, ensure the final mask is applied only once at the end.
    *   Use a stronger algorithm (e.g., CRC32, HMAC) instead of a simple linear combination with a fixed key.

2.  **Fix the State Machine:**
    *   Ensure the state machine consumes the delimiter bytes. Currently, if `b == 0x7B` occurs in the `HEADER` state, it does nothing. It should likely transition to `BODY` or reject the input.
    *   Validate that the input ends exactly after the `DONE` state.

3.  **Input Validation:**
    *   Ensure `len(key) > 0` before entering the loop.
    *   Ensure `len(input) > 0`.

4.  **Privileged Operation Guard:**
    *   The `privileged_operation` should ideally take the parsed payload (the `BODY` content) as an argument, not the raw `input`, to prevent leaking the delimiter bytes or key information during the operation.
```

### 1. Vulnerability Analysis

#### A. Buffer Overflow (Primary Critical Risk)
*   **Location:** `while (i <= payload_len)` loop.
*   **Mechanism:**
    *   The destination buffer `tmp` is fixed at **64 bytes**.
    *   The source data is copied from `buf` based on `payload_len`.
    *   The loop condition `i <= payload_len` allows the index `i` to reach `payload_len`.
    *   If `payload_len` is **64 or greater**, the loop writes to `tmp[64]` and beyond.
    *   Since C arrays are 0-indexed, valid indices are `0` to `63`. Writing to `tmp[64]` overflows the stack buffer.
*   **Trigger:** An attacker controls `buf` and sets the integer at `buf[4]` (`payload_len`) to a value $\ge 64$.
*   **Impact:**
    *   **Stack Corruption:** Overwrites the return address of `parse_packet` or adjacent stack variables (e.g., `len`, `tmp`'s metadata).
    *   **Return-Oriented Programming (ROP):** If the return address is overwritten, the function returns to attacker-controlled code.
    *   **Information Disclosure:** Overwrites local variables or function arguments, potentially leaking sensitive data.

#### B. Secondary Buffer Overflow (Conditional)
*   **Location:** `memmove(buf, buf + 12, payload_len + 32)`.
*   **Mechanism:**
    *   This line attempts to shift data within `buf`.
    *   If `payload_len` is large (e.g., $\ge 256$), `payload_len + 32` exceeds the likely allocated size of `buf` (assuming `buf` is a fixed-size buffer passed by the caller, or if `buf` is dynamically allocated but small).
    *   Even if `buf` is large, this operation is logically suspicious because it shifts data *forward* by `payload_len + 32`. If `payload_len` is huge, this writes far beyond the original buffer bounds, corrupting heap memory or adjacent stack frames if `buf` is on the stack.
*   **Trigger:** `payload_len` is set to a large value (e.g., 200+).

#### C. Logic Flow & Control Flow Hijack
*   **Flow:**
    1.  Validate length (`len < 8`).
    2.  Extract length (`payload_len`).
    3.  **Check:** `payload_len > len` (Returns -2).
    4.  **Copy:** Loop `i` from 0 to `payload_len`.
    5.  **Check:** `tmp[0] == 'A'` (Prints).
    6.  **Check:** `buf[1] == 0x42` (Shifts).
*   **Risk:** The logic assumes `payload_len` is trustworthy. The check `payload_len > len` prevents reading *past the end of the provided buffer*, but it **does not** prevent reading *past the destination buffer size* (`tmp`). The overflow happens because the destination size (64) is smaller than the maximum allowed `payload_len` (which is capped only by `len`).

### 2. Exploit Path Construction

An attacker can craft a packet `buf` to trigger the stack-based buffer overflow:

1.  **Packet Structure:**
    *   Bytes 0-3: Padding (ignored).
    *   Byte 4-7: Integer `payload_len`.
    *   Bytes 8+: Payload data.
2.  **Payload Construction:**
    *   Set `buf[4]` to `0x00000040` (64 in decimal).
    *   Fill `buf[8]` through `buf[71]` with data (e.g., `'A'` followed by junk).
    *   Ensure `len` is large enough (e.g., 1000) to pass the `len < 8` and `payload_len > len` checks.
3.  **Execution:**
    *   `payload_len` becomes 64.
    *   Loop runs for `i = 0` to `64`.
    *   `tmp[64]` is written.
    *   `tmp[65]` is written.
    *   ...
    *   `tmp[64]` overwrites the saved return address on the stack.
4.  **Result:**
    *   `parse_packet` returns to an address controlled by the attacker.
    *   If `tmp[0]` is `'A'`, `printf(tmp)` executes, potentially causing a format string vulnerability if `tmp` contains format specifiers (e.g., `%s`, `%x`), leading to arbitrary memory read/write.

### 3. Mitigation Guidance

#### Immediate Fixes (Code Level)
1.  **Fix the Loop Condition:**
    Change `i <= payload_len` to `i < 64`.
    ```c
    // Vulnerable
    while (i <= payload_len) { ... }
    
    // Fixed
    while (i < 64) { 
        if (i + 8 + 64 > len) break; // Safety check for source bounds
        tmp[i] = buf[8 + i];
        i++;
    }
    ```

2.  **Sanitize `payload_len`:**

    ```c
    int safe_len = payload_len;

    // Use safe_len in the loop
    while (i <= safe_len) { ... }
    ```

3.  **Fix `memmove` Bounds:**
    Ensure the destination and source pointers are valid relative to the buffer size.
    ```c
    // Check if destination (buf) + length exceeds buffer size
    if (payload_len + 32 > len) {

        return -3; 
    }
    memmove(buf, buf + 12, payload_len + 32);
    ```

1.  **Stack Canaries:** Enable compiler flags (e.g., `-fstack-protector-strong` in GCC/Clang) to detect stack corruption before the return address is overwritten.
2.  **ASLR (Address Space Layout Randomization):** Randomize stack and heap addresses to make ROP chains harder to construct.
3.  **Non-executable Stack (NX):** Prevent execution of injected shellcode on the stack.
4.  **Bounds Checking:** Use safe string/copy functions (`strncpy`, `memcpy` with explicit length limits) instead of manual loops or unchecked `memmove`.

```python
        Register a new user.

            password: Plain text password.

            The created User object.
            
        Raises:
            InvalidCredentialsError: If password is too weak.
            UserNotFoundError: If username already exists (if checking logic added).

        Returns:
            The authenticated User object.
            
        Raises:
            InvalidCredentialsError: If credentials do not match.
            UserNotFoundError: If user does not exist.

        user = self.storage.get_user(username)
        if not user:
            raise UserNotFoundError("User not found.")

        try:
            if not self._hasher.verify(user.password_hash, password):

        except VerifyError:

            user: The authenticated User object.
            
        Returns:
            AuthToken object containing the token and expiry.

            "username": user.username,

        return auth_token

            TokenExpiredError: If token is expired.

            return None
            
        if user.id != payload.get("sub"):
```

```python
class AuthManager:
    """
    Central orchestrator for authentication logic.
    
    Responsibilities:
    - Validate credentials against stored hashes.
    - Generate and validate tokens.
    - Manage user sessions.
    """

    def __init__(self, storage: Optional[StorageBackend] = None, secret_key: Optional[str] = None):
        self.storage = storage or InMemoryStorage()
        self.secret_key = secret_key or TOKEN_SECRET_KEY
        self._hasher = HASHER

    def register(self, username: str, email: str, password: str) -> User:
        """
        Register a new user.
        
        Args:
            username: Unique username.
            email: User email address.
            password: Plain text password.
            
        Returns:
            The created User object.
            
        Raises:
            InvalidCredentialsError: If password is too weak.
            UserNotFoundError: If username already exists (if checking logic added).
        """
        if len(password) < 8:
            raise InvalidCredentialsError("Password must be at least 8 characters.")
        
        # Hash the password
        password_hash = self._hasher.hash(password)
        
        user = User(
            id=secrets.token_hex(16),
            username=username,
            email=email,
            password_hash=password_hash
        )
        
        self.storage.save_user(user)
        return user

    def authenticate(self, username: str, password: str) -> User:
        """
        Authenticate a user by verifying credentials.
        
        Args:
            username: The username to look up.
            password: The plain text password provided by the user.
            
        Returns:
            The authenticated User object.
            
        Raises:
            InvalidCredentialsError: If credentials do not match.
            UserNotFoundError: If user does not exist.
        """
        user = self.storage.get_user(username)
        if not user:
            raise UserNotFoundError("User not found.")
        
        # Verify password against hash
        try:
            if not self._hasher.verify(user.password_hash, password):
                raise InvalidCredentialsError("Invalid credentials.")
        except VerifyError:
            raise InvalidCredentialsError("Invalid credentials.")
            
        return user

    def generate_token(self, user: User) -> AuthToken:
        """
        Generate a JWT token for the authenticated user.
        
        Args:
            user: The authenticated User object.
            
        Returns:
            AuthToken object containing the token and expiry.
        """
        payload = {
            "sub": user.id,
            "username": user.username,
            "exp": int((datetime.now(timezone.utc) + TOKEN_EXPIRY).timestamp())
        }
        
        token = _encode_token(payload, self.secret_key)
        
        auth_token = AuthToken(
            token=token,
            expires_at=datetime.fromtimestamp(payload["exp"], tz=timezone.utc),
            user_id=user.id
        )
        
        self.storage.save_token(auth_token)
        return auth_token

    def validate_token(self, token: str) -> Optional[User]:
        """
        Validate a token and return the associated user.
        
        Args:
            token: The JWT string to validate.
            
        Returns:
            The User object if valid, None otherwise.
            
        Raises:
            TokenExpiredError: If token is expired.
            TokenGenerationError: If token is malformed.
        """
        payload = _decode_token(token, self.secret_key)
        user = self.storage.get_user(payload.get("username"))
        
        if not user:
            return None
            
        if user.id != payload.get("sub"):
            return None
            
        return user
```

### Vulnerability-Oriented Reverse Engineering Assessment

The provided authentication snippet contains **critical, show-stopping security flaws** that render the system completely insecure. This code exhibits patterns typical of "tutorial hell" implementations that are dangerous in production.

#### 1. Cryptographic Primitive Failure (CRITICAL)
*   **Flaw:** `sha1(password).hexdigest()`
*   **Analysis:** SHA-1 is cryptographically broken and unsuitable for password hashing. It is a fast, non-iterative hash function.
*   **Impact:**
    *   **Rainbow Table Attacks:** Since SHA-1 is fast, attackers can pre-compute hashes for common passwords (rainbow tables) and crack them instantly.
    *   **Brute Force:** Attackers can test billions of password guesses per second on a single server.
    *   **Reversibility:** SHA-1 is deterministic. If an attacker obtains the database, they can reverse the hash (or crack it) to recover the original password.
*   **Requirement:** Must use **Argon2** (preferred) or **bcrypt** / **scrypt**. These are slow, memory-hard functions designed specifically to resist brute-force attacks.

#### 2. Token Secret Exposure (CRITICAL)
*   **Flaw:** `token = f'{username}:{int(time.time())}'`
*   **Analysis:** The token is constructed as a plain string concatenation of the username and a timestamp.
*   **Impact:**
    *   **Trivial Forgery:** An attacker who knows a valid username can generate a valid token for that user at any time they choose. They do not need to steal a secret key or intercept a network packet.
    *   **No Signature:** There is no cryptographic signature (HMAC/JWT) to verify the token's integrity. An attacker can modify the token payload (e.g., change `username` to `admin`) and the server will accept it.
    *   **No Expiry Enforcement:** The code stores an expiry time, but the token itself does not contain a signature. The server cannot cryptographically verify that the token belongs to the user or hasn't been tampered with.
*   **Requirement:** Tokens must be signed (e.g., JWT with a secret key) or verified via HMAC. The token must be opaque and unguessable.

#### 3. Race Condition / Timing Attack (HIGH)
*   **Flaw:** `if sha1(password).hexdigest() != user.password_hash:`
*   **Analysis:** The code compares the hash of the input password against the stored hash using a string comparison.
*   **Impact:**
    *   **Timing Attacks:** If the comparison is not constant-time (which string comparison usually isn't), an attacker can measure the response time to determine how many characters of the password are correct. This allows them to guess the password character by character.
    *   **Database Enumeration:** The `db.find_user(username)` call leaks information. If the user exists, the server takes time $T_1$. If the user does not exist, it takes time $T_2$. An attacker can determine valid usernames by measuring latency.
*   **Requirement:** Password verification must be done in **constant time**. Username lookup should be decoupled from password verification (e.g., return "Invalid credentials" for both valid/invalid users to prevent enumeration).

#### 4. In-Memory Token Store (HIGH)
*   **Flaw:** `token_cache[token] = ...`
*   **Analysis:** The code assumes a dictionary (`token_cache`) is used for storage.
*   **Impact:**
    *   **Persistence:** If the application restarts, the `token_cache` is lost. Users are logged out.
    *   **Memory Limits:** The cache will grow indefinitely as users log in, eventually crashing the server.
    *   **Concurrency:** If multiple processes are running, a standard dictionary is not shared, leading to race conditions where tokens are not validated or expired correctly.
*   **Requirement:** Use a persistent, distributed store like **Redis** with TTL (Time To Live) support.

#### 5. Lack of Input Validation (MEDIUM)
*   **Flaw:** `username` and `password` are passed directly.
*   **Analysis:** No sanitization or length checks.
*   **Impact:**
    *   **Injection:** If `username` is used in a SQL query later, it is vulnerable to SQL Injection.
    *   **Denial of Service:** An attacker can send a massive password string, causing the SHA-1 hash to take longer, potentially DoS-ing the server.
*   **Requirement:** Validate input length, type, and sanitize against injection.
