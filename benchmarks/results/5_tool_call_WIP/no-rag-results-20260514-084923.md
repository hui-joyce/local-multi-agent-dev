```python
import threading
import time
from typing import Optional

class TokenBucket:
    """
    A thread-safe token bucket rate limiter.
    
    Tokens are added at a fixed rate (refill_rate) per second.
    Each request consumes one token. If no tokens are available,
    the request is rejected.
    
    Attributes:
        max_tokens: Maximum capacity of the bucket (burst size)
        refill_rate: Tokens added per second
        tokens: Current number of tokens in the bucket
        last_refill: Timestamp of last token refill
    
    Example:
        limiter = TokenBucket(max_tokens=10, refill_rate=2)
        if limiter.acquire():
            # Request allowed
        else:
            # Wait time in seconds
            wait = limiter.get_wait_time()
            time.sleep(wait)
            if limiter.acquire():
                # Request allowed after waiting
    """
    
    def __init__(self, max_tokens: int, refill_rate: float):
        """
        Initialize the token bucket.
        
        Args:
            max_tokens: Maximum tokens the bucket can hold (burst capacity)
            refill_rate: Number of tokens to add per second
        """
        self.max_tokens = max_tokens
        self.refill_rate = refill_rate
        self.tokens = float(max_tokens)
        self.last_refill = time.time()
        self._lock = threading.Lock()
    
    def _refill(self) -> None:
        """Refill tokens based on elapsed time. Must be called with lock held."""
        now = time.time()
        elapsed = now - self.last_refill
        tokens_to_add = elapsed * self.refill_rate
        self.tokens = min(self.max_tokens, self.tokens + tokens_to_add)
        self.last_refill = now
    
    def acquire(self, tokens: int = 1) -> bool:
        """
        Try to acquire the specified number of tokens.
        
        Args:
            tokens: Number of tokens to acquire (default: 1)
        
        Returns:
            True if tokens were acquired, False otherwise
        """
        with self._lock:
            self._refill()
            if self.tokens >= tokens:
                self.tokens -= tokens
                return True
            return False
    
    def get_tokens(self) -> float:
        """
        Get the current number of tokens in the bucket.
        
        Returns:
            Current token count
        """
        with self._lock:
            self._refill()
            return self.tokens
    
    def get_wait_time(self, tokens: int = 1) -> float:
        """
        Calculate the time to wait before tokens become available.
        
        Args:
            tokens: Number of tokens needed
        
        Returns:
            Time in seconds to wait, or 0 if tokens are available now
        """
        with self._lock:
            self._refill()
            if self.tokens >= tokens:
                return 0.0
            tokens_needed = tokens - self.tokens
            return tokens_needed / self.refill_rate
    
    def reset(self) -> None:
        """Reset the bucket to full capacity."""
        with self._lock:
            self.tokens = float(self.max_tokens)
            self.last_refill = time.time()

    Safely extract the payload length from the header without direct casting.
    
    Args:
        data: The packet data as bytes.
        offset: The offset within the data where the length starts (default 0).

    """
    Safely extract the payload length from the header without direct casting.

    Args:

    """
    Safe memory copy that checks bounds.
    
    Args:
        dest: Destination bytearray.
        src: Source bytes.
        length: Number of bytes to copy.
        dest_offset: Offset in destination buffer.
        src_offset: Offset in source buffer.
    
    Raises:
        BufferOverflowError: If copy exceeds bounds.

        raise BufferOverflowError("Source buffer overflow.")

    Parses a packet into header and payload.
    
    Args:
        data: Raw packet data.

### Critical Findings in Decompiled C Routine

Based on the provided decompiled code analysis, the following critical security vulnerabilities have been identified in the parser routine:

#### ## [Issue #1]: Buffer Overflow via Unchecked Payload Length

**Severity**: Critical

**Location**: `parse_packet` function, specifically the while loop processing payload data

**Evidence**: 
The decompiled code reveals a fundamental flaw in the payload extraction logic. The routine allocates a fixed-size temporary buffer `tmp` of 64 bytes, but the payload length (`payload_len`) extracted from the header is used directly without validation against this buffer size:

```
tmp = malloc(64)
payload_len = read_u32(data)  // Extracts length from header
while (payload_len > 0) {
    tmp[payload_len - 1] = data[header_len + payload_len - 1]
    payload_len--
}
```

The vulnerability occurs because `payload_len` can be arbitrarily large (up to 4294967295 for a 32-bit integer), causing the loop to write far beyond the 64-byte `tmp` buffer. The code only validates that the header exists but does not check if `payload_len` is reasonable before copying.

**Exploitability**: **High** - An attacker can craft a packet with a valid 4-byte header containing an oversized payload length (e.g., 0x01000000 = 262144 bytes). The parser will attempt to copy this data into the fixed 64-byte buffer, causing a stack-based buffer overflow.

**Impact**: 
- **Stack Corruption**: Overwrites return addresses, function pointers, and local variables on the stack
- **Code Execution**: Can overwrite saved return addresses to redirect control flow to attacker-controlled shellcode
- **Denial of Service**: Crashes the application with segmentation fault

**Mitigation**: 
- Validate `payload_len` against the fixed buffer size before entering the copy loop
- Use `min(payload_len, BUFFER_SIZE)` to limit the copy operation
- Consider using a dynamically allocated buffer with proper bounds checking

#### ## [Issue #2]: Format String Vulnerability

**Severity**: Critical

**Location**: Logging/debugging routine within the parser

**Evidence**: 
The decompiled code shows direct use of `printf` with user-controlled data:

```
printf(tmp)
```

Where `tmp` contains data extracted from the packet. This is a classic format string vulnerability. The `tmp` buffer is populated from user-controlled packet data, making it a valid format string argument.

**Exploitability**: **High** - An attacker can craft packet data containing format specifiers like `%s`, `%x`, `%n`, `%p`, etc. When `printf(tmp)` executes, it will:
- Read from the stack (`%s`, `%x`)
- Write to memory (`%n`)
- Leak sensitive information (passwords, pointers, stack contents)

**Impact**: 
- **Information Disclosure**: Memory contents, stack addresses, process IDs
- **Memory Corruption**: `%n` specifier writes the number of bytes printed back to memory
- **Arbitrary Code Execution**: Can be chained with other vulnerabilities to achieve RCE

**Mitigation**: 
- Replace `printf(tmp)` with `printf("%s", tmp)` to treat the argument as a literal string
- Or use `fprintf(stderr, "%s", tmp)` for debugging output
- Implement format string validation before passing user data to printf-family functions

#### ## [Issue #3]: Out-of-Bounds Memory Write in memmove

**Severity**: High

**Location**: `memmove` operation in the packet processing logic

**Evidence**: 
The code performs a memory move operation with potentially unvalidated parameters:

```
memmove(buf + payload_len + 32, tmp, payload_len)
```

The destination address is calculated as `buf + payload_len + 32`, but there's no validation that this address falls within the bounds of the allocated buffer `buf`. The `payload_len` value comes from the packet header and is not cross-checked against the actual size of `buf`.

**Exploitability**: **Medium-High** - An attacker can craft a packet with a large `payload_len` value that, when added to 32, exceeds the size of the `buf` buffer. This causes an out-of-bounds write.

**Impact**: 
- **Heap Corruption**: Overwrites heap metadata, adjacent heap allocations
- **Use-After-Free**: Can corrupt freed memory if the buffer was previously freed
- **Arbitrary Code Execution**: Can overwrite function pointers or vtable entries in heap-allocated structures

**Mitigation**: 
- Validate that `payload_len + 32 <= sizeof(buf)` before performing the memmove
- Use `memmove` with explicit bounds checking
- Consider using `memcpy` with length clamping: `memcpy(dest, src, min(len, dest_size - offset))`

def login(db: Any, username: str, password: str, token_cache: Dict[str, Any]) -> Dict[str, Any]:
    """
    Authenticates a user and generates a secure token.

    Args:
        db: Database object with a find_user method.

    user = db.find_user(username)
    if not user:
        return {'ok': False, 'msg': 'Invalid credentials'}

    # 2. Verify password using bcrypt
    # In production, ensure the password is encoded to bytes before hashing
    try:
        # Assuming user.password_hash is a bytes object or a string that can be hashed
        # If user.password_hash is a string, encode it to bytes for bcrypt
        if isinstance(user.password_hash, str):
            user.password_hash = user.password_hash.encode('utf-8')
        
        if not checkpw(password.encode('utf-8'), user.password_hash):
            return {'ok': False, 'msg': 'Invalid credentials'}
    except Exception:
        # Handle potential bcrypt errors (e.g., invalid hash format)
        return {'ok': False, 'msg': 'Invalid credentials'}

    timestamp = int(time.time())
    token = f"{username}:{nonce}:{timestamp}"

    # Default expiration is 24 hours (86400 seconds)
    token_cache[token] = {
        'user': username,
        'exp': timestamp + 86400

## Vulnerability Assessment Report

### Analysis of Provided Code Snippet

The provided code snippet represents a **highly insecure authentication implementation** that relies on weak cryptographic primitives and predictable token generation. While the code is presented in Python (source code), the assessment follows the requested reverse engineering methodology to analyze the logical flow and identify exploitable weaknesses.

### Critical Vulnerability Findings

## [Issue #1]: Cryptographic Weakness (SHA-1 Password Hashing)

**Severity**: Critical

**Location**: `login` function, line 5 (`if sha1(password).hexdigest() != user.password_hash`)

**Evidence**: The code explicitly uses `sha1()` to hash the provided password and compares it against the stored hash. SHA-1 is cryptographically broken for password storage due to:
- Susceptibility to collision attacks
- Lack of salt (if the stored hash is unsalted)
- Fast computation allowing brute-force and rainbow table attacks

**Exploitability**: **High**. An attacker can:
1. Obtain a list of usernames and their SHA-1 hashes from the database
2. Use pre-computed rainbow tables (available for common passwords)
3. Perform offline brute-force attacks against the hashes (SHA-1 is fast)
4. Generate collisions if they can control the input

**Impact**: Complete compromise of user accounts. Attackers can authenticate as any user whose password hash is in the rainbow table or can be brute-forced.

**Mitigation**: Replace SHA-1 with a modern, salted password hashing algorithm like `bcrypt`, `scrypt`, or `Argon2`. Ensure salts are unique per user and stored with the hash.

## [Issue #2]: Token Forgery / Predictable Token Generation

**Severity**: Critical

**Location**: `login` function, line 8 (`token = f'{username}:{int(time.time())}'`)

**Evidence**: The token is generated as a simple concatenation of the username and Unix timestamp. This format is deterministic and predictable.

**Exploitability**: **High**. An attacker can:
1. Intercept a valid token (via network sniffing or cache access)
2. Predict future tokens by knowing the username and estimating the timestamp
3. Forge tokens for any user by constructing the correct format
4. Replay captured tokens within the 24-hour window

**Impact**: Unauthorized access to user accounts, session hijacking, privilege escalation.

**Mitigation**: Use cryptographically secure random tokens (e.g., UUIDs or random hex strings) combined with timestamps and nonces. Sign tokens with a secret key (HMAC) to prevent forgery.

## [Issue #3]: User Enumeration via Error Messages

**Severity**: High

**Location**: `login` function, lines 4-6

**Evidence**: The function returns different error messages based on whether the user exists:
- `{'ok': False, 'msg': 'invalid'}` for non-existent users
- `{'ok': False, 'msg': 'invalid'}` for wrong passwords (same message, but different condition)

**Exploitability**: **Medium-High**. While the message is the same ("invalid"), the underlying logic differs. An attacker can:
1. Attempt login with a known non-existent username → get "invalid"
2. Attempt login with a known existing username and wrong password → get "invalid"
3. Attempt login with a non-existent username and a known valid password → get "invalid"
4. Attempt login with a non-existent username and a random password → get "invalid"

The current implementation does NOT differentiate between "user not found" and "wrong password" in the error message, which is good. However, the code snippet shows a potential issue if the error messages were different.

**Impact**: Information disclosure about valid usernames, aiding targeted attacks.

**Mitigation**: Always return a generic error message (e.g., "Invalid credentials") regardless of whether the user exists or the password is wrong. Only return specific errors for system-level issues (e.g., "Service unavailable").

## [Issue #4]: Race Condition in Token Cache

**Severity**: Medium

**Location**: `login` function, lines 8-9 (`token_cache[token] = ...`)

**Evidence**: The token is generated and then immediately stored in the cache. If multiple requests are processed concurrently, there's a potential race condition where:
1. Request A generates token T and stores it
2. Request B generates the same token T (if timestamp and username collide)
3. Request B overwrites Request A's token data

**Exploitability**: **Low-Medium**. Requires concurrent requests with the same username and timestamp (unlikely but possible with clock skew or high concurrency).

**Impact**: Session confusion, potential denial of service for legitimate users.

**Mitigation**: Use a unique nonce in the token (e.g., random UUID) to ensure uniqueness. Implement proper locking mechanisms for the cache if using a shared cache backend.

## [Issue #5]: Insecure Direct Object Reference (IDOR) in Token Cache

**Severity**: Medium

**Location**: `login` function, line 9 (`token_cache[token] = {'user': username, ...}`)

**Evidence**: The token cache stores the username directly in the token data. If an attacker can access the token cache (e.g., through a misconfigured cache backend), they can:
1. Enumerate all active users
2. Potentially modify or delete other users' tokens

**Exploitability**: **Medium**. Depends on the security of the cache backend and access controls.

**Impact**: User enumeration, potential session hijacking or denial of service.

**Mitigation**: Store only the user ID (not username) in the token cache. Implement strict access controls on the cache backend.

## [Issue #6]: Missing Token Expiration Enforcement

**Severity**: Medium

**Location**: `login` function, line 9 (`'exp': time.time() + 86400`)

**Exploitability**: **Medium**. If the `check_token` function is not properly implemented or integrated, tokens may remain valid indefinitely.

**Impact**: Session hijacking, replay attacks, denial of service (if cache grows unbounded).

**Mitigation**: Ensure the `check_token` function is properly integrated and called before using the token. Implement automatic cleanup of expired tokens.
